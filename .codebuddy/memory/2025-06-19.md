- `_build_graph_summary` 改用独立函数 `build_graph_context`

**前端 `chatStore.js`**：
- `fetchGraph()` 在边数据中新增 `edgeId` 字段（数据库主键 ID）

**前端 `ForceGraph.vue`**：

## 知识图谱创建过程逻辑矛盾修复（第二轮审查）

### 发现的逻辑矛盾

1. **`process_message` 返回空图谱分析结果**：`return reply, mode, {"suggestions": [], "applied": [], "pending": []}` 硬编码空结果，但 `_analyze_and_apply` 才是真正执行分析的地方。

2. **`_analyze_and_apply` 与调用方共享 kg 实例导致 use-after-close 风险**：
   - `process_message` 在 `kg.close()` 后，后台任务 `_analyze_and_apply` 仍在使用同一个 kg
   - 虽然之前注释说"后台任务会创建自己的 kg"，但代码实际并未这样做

3. **`create_node_from_ai` 中 `created_at` 硬编码为空字符串**：`"created_at": ""` 覆盖了 `add_node()` 的默认值 `datetime.now().isoformat()`

4. **`_apply_suggestion` 中 `recommended_edges` 匹配逻辑不完整**：只处理了 `to==新节点ID` 的情况，遗漏了 `from==新节点ID`（新节点是已有节点的前置）

5. **前端 `_index` 已无意义但仍保留**：边操作改用 `edgeId` 后，`_index` 在任何地方都不再被使用

### 修复内容

**后端 `chat_service.py`**：
- `_analyze_and_apply` 参数从 `kg: KnowledgeGraph` 改为 `user_id: int`，内部自己创建和管理 kg 生命周期
- 新增 `finally: kg.close()` 确保资源释放
- 有自动应用的变更时主动 `publish("graph_updated")`
- `process_message` 和 `process_background_tools` 调用处传入 `user_id` 而非 `kg`

**后端 `api/v1/knowledge.py`**：
- `create_node_from_ai` 移除 `"created_at": ""`，让 `add_node()` 使用默认值
- `_apply_suggestion` 中 `recommended_edges` 匹配逻辑增加注释说明方向语义

**前端 `chatStore.js`**：
- 移除 `_index` 字段（不再需要）

**前端 `ForceGraph.vue`**：
- `renderGraph()` 移除 `_index` 字段
- 更新注释

## 知识图谱视觉效果 —— Obsidian 极简风格重写

### 设计理念
去装饰化：纯色节点 + 细线边，无光晕/渐变/星星/进度环，像 Obsidian 图谱一样干净。

### 核心变更

**节点**（`NODE_RADIUS = 16`）：
- 仅一个 circle：纯色填充 + 1px 同色细描边（`stroke-opacity: 0.3`）
- 颜色：未学习 → `--color-graph-node` 灰，已学习 → `--color-accent` 主题色
- 透明度根据 mastery 微调（0→0.6, 100→1.0）
- 标签在节点右侧（`dx: radius+8, text-anchor: start`），Obsidian 风格右置
- 悬停：放大 4px + 主题色描边 2px + 非关联节点淡化到 0.12

**边**：
- `stroke-width: 1.2, stroke-opacity: 0.4` 极细半透明
- 悬停关联边变主题色，非关联边淡化到 0.12
- 箭头 4px 极简三角

**力场**：
- `forceLink: distance(140), strength(0.3)` 更松散自然
- `velocityDecay: 0.35` 更高摩擦
- `collision: radius(28), strength(0.6)`
- 拖拽 alphaTarget: 0.12

**移除**：
- `masteryGlow()`, `masteryArc()`, `difficultyStars()` 全部删除
- `.node-glow`, `.node-highlight`, `.node-progress-ring`, `.node-difficulty` 全部移除
- 悬停时不再操作这些不存在的元素
- 更新提示去除 emoji（🔄 → 纯文字）

## 知识图谱语义关系验证强化

### 问题
创建知识节点和边时，系统完全信任 LLM 的判断，代码层面没有任何语义验证：
- `add_edge()` 只做去重检查，不验证两个节点是否真的有逻辑关系
- GraphAnalyzer 提示词约束太弱，没有明确要求语义判断
- LLM function calling 的 `add_edge` 工具描述过于简单
- 可能出现：无关节点被连边、自环边、循环前置依赖、低置信度边被自动应用

### 修复（四层防御）

**第1层 — 提示词强化 (`graph_analyzer.py`)**：
- 新增「核心原则：宁可漏过，不可乱连」
- 为 4 种关系类型各写详细语义判断标准（含定义、判断标准、反例、关键问题）
- 新增高置信度要求：add_edge 必须在确信有明确知识关系时才 confidence >= 0.7
- 模糊/牵强关系 confidence 必须 <= 0.4
- recommended_edges 中每条边要求 reason 解释为何确实存在该关系

**第2层 — 工具描述强化 (`llm_client.py`)**：
- `add_edge` 工具 description 加入 ⚠️ 警告：仅在确认实质性知识关系时才调用
- `relation` 参数描述细化每种关系的语义
- `label` 要求具体说明为何有该关系，而非仅仅"相关"
- `add_knowledge_node` 的 `from_nodes` 加入 ⚠️ 警告：只能填真正的前置知识节点

**第3层 — 代码验证层 (`graph_analyzer.py`)**：
- `_validate_and_filter()` 新增：
  - `add_edge` 置信度门槛 0.7（低于此值直接丢弃）
  - `recommended_edges` 置信度门槛 0.5
  - 自环边检测（from == to 丢弃）
  - relation 类型白名单校验
- 所有被过滤的边都有 `logger.info` 日志记录

**第4层 — 数据层约束 (`knowledge_graph.py`)**：
- `add_edge()` 新增：
  - 自环边拒绝（from == to）
  - relation 枚举值校验
  - 循环前置依赖检测：`_has_path()` BFS 搜索 prerequisite 路径，防止形成环
- 新增 `_has_path(from_id, to_id)` 方法：检查是否存在 prerequisite 路径

**第5层 — 行为提示词 (`chat_service.py`)**：
- `TOOL_CAPABILITY_PROMPT` 新增 ⚠️ 原则：宁可漏掉一条关系，也不要创建错误的关系
- 关联节点和扩展图谱的能力说明都加入语义判断提醒

## AI 权限控制：不能修改人类创建的内容

### 问题
`added_by` 只是一个标记字段，AI 在执行工具调用时完全不检查它：
- AI 可以 `update_node_content` / `update_node_info` / `delete_node` 任意节点
- AI 可以在两个人类节点之间创建 `prerequisite` 边（影响学习路径）
- AI 可以删除人类创建的边
- `update_mastery` 会把 `added_by` 覆盖为 "ai"

### 修复（`knowledge_graph.py` 核心 + 调用方透传）

**新增两个守卫方法**：
- `_guard_human_content(caller, node_id)`: caller="ai" 时检查节点 added_by 是否为 "human"，是则抛 PermissionError
- `_guard_human_edge(caller, edge_id)`: 同理检查边

**受保护的写操作（均新增 `caller` 参数）**：
| 方法 | AI 限制 |
|------|---------|
| `update_node_content(node_id, content, mode, caller)` | 不能修改 human 节点 |
| `update_node_info(node_id, data, caller)` | 不能修改 human 节点 |
| `remove_node(node_id, caller)` | 不能删除 human 节点 |
| `remove_edge_by_id(edge_id, caller)` | 不能删除 human 创建的边 |
| `add_edge(edge_data, caller)` | 不能在两个 human 节点间加 prerequisite |

**调用方透传 `caller="ai"`**：
- `llm_client.py` execute_kg_tool(): 所有工具调用传 caller="ai"
- `llm_client.py` 新增 `PermissionError` 捕获，返回友好提示让 AI 知道权限不足
- `knowledge.py` _apply_suggestion(): 传 caller="ai"
- `chat_service.py` _analyze_and_apply(): PermissionError 被捕获并移到 pending_list

**行为提示词 (`chat_service.py` TOOL_CAPABILITY_PROMPT)**：
- 新增「权限限制（严格执行）」部分，明确告知 AI：
  - 不能修改/删除人类创建的节点
  - 不能在两个人类节点间加 prerequisite
  - 可以在人类节点间加 related/confusion/extension（但标记为 AI 建议）
  - 自己创建的节点可以自由修改

**人类 API 不受影响**：所有前端 HTTP 请求默认 caller="human"，守卫不触发。

## 修复：create_node_from_ai 缺少 caller="ai" + recommended_edges 处理不完整

### 问题
1. `create_node_from_ai` 内部调用 `kg.add_edge()` 创建 prerequisite 边时没传 `caller="ai"`，导致边权限检查不触发（实际默认 `"human"`）
2. `_apply_suggestion` 的 `add_node` action 中，`recommended_edges` 只处理 `to==新节点` 的 prerequisite 边，`from==新节点` 的反向边和 non-prerequisite 边被完全遗漏
3. 缺少 `PermissionError` 捕获

### 修复
- `create_node_from_ai`: `kg.add_edge({...}, caller="ai")`，并新增 `except PermissionError: pass`
- `_apply_suggestion` `add_node` action: 重构 `recommended_edges` 处理逻辑
  - `prerequisite` + `to==新节点` → from_nodes（前置）
  - 其他 relation 类型 + `to==新节点` → extra_edges（独立创建）
  - `from==新节点` → extra_edges（独立创建）
  - 创建节点后，遍历 extra_edges 创建非 prerequisite 边（caller="ai"）
